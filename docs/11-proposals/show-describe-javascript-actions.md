# Proposal: SHOW/DESCRIBE JavaScript Actions

## Overview

**Document type:** `JavaScriptActions$JavaScriptAction`
**Prevalence:** 283 across test projects (96 Enquiries, 81 Evora, 106 Lato)
**Priority:** High — present in every project, used heavily for nanoflow logic

JavaScript Actions are the nanoflow equivalent of Java Actions. They define callable functions with typed parameters and return types, targeting Web, Native, or All platforms. The JavaScript source code is stored on disk alongside the project.

## Relationship to Java Actions

JavaScript Actions and Java Actions share the **same `CodeActions$` type system** for parameters and return types. The key differences are:

| Aspect | Java Action | JavaScript Action |
|--------|------------|-------------------|
| BSON type | `JavaActions$JavaAction` | `JavaScriptActions$JavaScriptAction` |
| Parameter type | `JavaActions$JavaActionParameter` | `JavaScriptActions$JavaScriptActionParameter` |
| Platform field | absent | `"All"` \| `"Web"` \| `"Native"` |
| Source location | `javasource/{module}/actions/{Name}.java` | `javascriptsource/{module}/actions/{Name}.js` |
| Source markers | `// BEGIN USER CODE` / `// END USER CODE` | Same markers, plus `// BEGIN EXTRA CODE` / `// END EXTRA CODE` |
| Nanoflow param | not available | `NanoflowJavaScriptActionParameterType` |
| Import/Export mapping params | `ImportMappingJavaActionParameterType`, `ExportMappingJavaActionParameterType` | not available |

The `CodeActions$` type hierarchy is shared and identical:
- Primitives: `BooleanType`, `IntegerType`, `LongType`, `DecimalType`, `FloatType`, `StringType`, `DateTimeType`, `VoidType`
- Entity types: `ConcreteEntityType` (specific entity), `ParameterizedEntityType` (type parameter ref)
- Collections: `ListType` (list of entities)
- Enumerations: `EnumerationType`

Parameter type wrappers differ slightly:
- Both: `BasicParameterType` (wraps a `Type`), `EntityTypeParameterType`, `StringTemplateParameterType`
- Java-only: `ImportMappingJavaActionParameterType`, `ExportMappingJavaActionParameterType`
- JS-only: `NanoflowJavaScriptActionParameterType` (pass a nanoflow as callback)

## JavaScript Source Code

Source files are stored on disk at:
```
{project_root}/javascriptsource/{modulename}/actions/{ActionName}.js
```

Note: `{modulename}` uses the **original module name casing** (not lowercased like `javasource/`).

Example JS source:
```javascript
// This file was generated by Mendix Studio Pro.
//
// WARNING: Only the following code will be retained when actions are regenerated:
// - the import list
// - the code between BEGIN USER CODE and END USER CODE
// - the code between BEGIN EXTRA CODE and END EXTRA CODE
// Other code you write will be lost the next time you deploy the project.
import { Big } from "big.js";

// BEGIN EXTRA CODE
import { someHelper } from "../utils/helpers";
// END EXTRA CODE

/**
 * Validates the input string
 * @param {string} input
 * @param {Big} maxLength
 * @returns {Promise.<boolean>}
 */
export async function ValidateInput(input, maxLength) {
    // BEGIN USER CODE
    if (!input || input.length === 0) {
        return false;
    }
    return input.length <= Number(maxLength);
    // END USER CODE
}
```

The DESCRIBE output should include both the USER CODE and EXTRA CODE sections.

## Proposed MDL Syntax

### SHOW JAVASCRIPT ACTIONS

```
SHOW JAVASCRIPT ACTIONS [IN Module]
```

Output table columns:

| Qualified Name | Module | Name | Platform | Return Type | Params | Exposed |
|----------------|--------|------|----------|-------------|--------|---------|

### DESCRIBE JAVASCRIPT ACTION

```
DESCRIBE JAVASCRIPT ACTION Module.Name
```

Output format — follows `DESCRIBE JAVA ACTION` syntax with inline parameters:

```
/**
 * Validates the input string
 */
CREATE JAVASCRIPT ACTION MyModule.ValidateInput(input: String NOT NULL, maxLength: Integer)
  PLATFORM Web
  RETURNS Boolean
  EXPOSED AS 'Validate Input' IN 'Validation'
AS $$
    if (!input || input.length === 0) {
        return false;
    }
    return input.length <= Number(maxLength);
$$;
-- EXTRA CODE:
-- import { someHelper } from "../utils/helpers";
```

For a JS action with type parameters and entity types (same syntax as Java Actions):

```
CREATE JAVASCRIPT ACTION MyModule.SyncEntities<T>(entityType: ENTITY <T>, items: LIST OF <T>)
  PLATFORM All
  RETURNS Boolean;
```

For a JS action with a nanoflow parameter (JS-specific):

```
CREATE JAVASCRIPT ACTION MyModule.WithRetry(action: Nanoflow, maxRetries: Integer)
  PLATFORM All
  RETURNS Boolean;
```

Key syntax points matching Java Actions:
- Parameters are **inline in parentheses** after the action name: `Name(param1: Type, param2: Type NOT NULL)`
- `NOT NULL` indicates required parameters (not a separate `REQUIRED` keyword)
- Type parameters use `<T>` syntax: `Name<T>(entityType: ENTITY <T>)`
- Source code uses `AS $$ ... $$` dollar-quoting

Key differences from Java Actions:
- `PLATFORM Web|Native|All` clause (absent for Java Actions)
- `Nanoflow` parameter type (JS-only)
- Extra code section shown as comments
